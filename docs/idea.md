このシステムは、個人が学習した内容（主にプログラミング言語理論・型理論・論理学・数学）を、記事として蓄積しつつ、同時に概念（用語）を定義単位で一元管理するための静的サイトである。読者は記事を読む途中で用語に遭遇した際、その場で意味を確認できる必要があり、そのために概念リンクと hover preview を提供する。記事としての読みやすさと、辞書的な参照性を同時に満たすことが目的である。

システムは GitHub を中心に運用される。ユーザーは posts（記事）および defs（定義）としてコンテンツを記述し、push を契機に GitHub Actions が起動する。ビルドでは frontmatter の status により公開対象（published）のみを選別し、Markdown をパースし、概念リンクを解決し、preview 用データおよび検索用インデックスを生成したうえで HTML を出力し、GitHub Pages にデプロイする。クライアントでは Astro + React による通常の描画を行い、hover 時に preview を表示し、MathJax により数式および導出図をレンダリングする。

ドメインは Post（記事）と Definition（定義）の 2 種類のみを持つ。Definition は 1 ページ 1 概念を基本とし、id・title・aliases・status を持つ。内容は「定義を記述する definition_block」と「補足説明の本文」から構成され、preview や検索、一覧表示はすべて definition_block を基準とする。異なる分野で意味が異なる場合は別 id として分離する。Post は通常の文章であり、定理・証明・局所定義などはページ種別としては扱わず、記事内の構造として自由に記述する。

記事中では [[term]] という記法により概念リンクを記述し、対応する Definition へのリンクと preview 表示の対象とする。ページ内リンクは通常の #anchor を使用し、記事同士のリンクは通常の Markdown を用いる。記事内のみで有効なローカル定義も許可し、これはページ外からは参照されず、記号導入や一時的な概念の説明に用いる。

preview は hover または長押しで表示され、Definition の definition_block をそのまま描画する。数式や導出図も含めて表示される必要があるため、ビルド時に preview 用のインデックスを生成し、クライアント側で該当データを取得して描画し、その後 MathJax を再実行する。

数式は TeX 記法で記述し、MathJax によりレンダリングする。導出図は bussproofs（prooftree）を使用し、MathJax の拡張で対応する。

UI はパステルカラーを基調とし、かわいい印象とターミナル風雰囲気を併せ持たせる。レイアウトは数学書のような整った組版を目指す。画面としては、ホームに最新記事とシリーズ一覧を表示し、各記事および定義ページではタイトル、本文、タグ、 backlink などを表示する。

メタデータは frontmatter で管理し、status（draft / published / scrap）、series、series_order などを持つ。tags は横断的分類として扱い、series は順序付きの記事群として扱う。検索は静的に実装し、title・aliases・tags・definition_block・記事冒頭を対象とする。

ビルド処理はコンパイラに近い構造で設計する。すなわち parser により入力を解析し、AST を生成し、link 解決や index 生成などの変換を行い、最終的に HTML を生成する codegen に至るパイプラインとする。各フェーズは分離し、ドメインロジック（定義・リンク・検索など）はフロントエンドから独立させ、テスト可能な構造とする。一方で表示部分は通常の Astro プロジェクトとして実装する。

初期段階では CMS、include 機能、曖昧な自動リンク、複雑な権限管理は導入しない。未決定事項として、開発の進め方、 Markdown 拡張仕様、AST 設計(本当に必要か)、preview index 形式、検索 index の詳細、UI の細部設計が残る。

開発手法はこれから検討。フロントのテスト等経験がないためベストプラクティスを探して実践したい。/home/kise/repos/compro-env に 僕たちの直近のプロジェクトがあるのと、一応 /home/kise/repos/docdd にフロントの SpecDD 的な参考レポジトリがある。 compro-env の /spec-\* コマンドの開発フローをまずコピーして、そこに docdd 内で使われている MCP たちと仕様の運用方法やテスト方法等の詳細を反映していき、この build+Astro なプロジェクトに合わせて環境を確立することから始めたい。
